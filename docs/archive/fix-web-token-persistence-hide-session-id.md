# Fix: persist Bearer token across refresh + drop session-id debug input

## Symptom
After a page refresh the left sidebar showed no chat history, even though the
conversation was persisted in Postgres.

## Root cause
Client-side identity, not the backend. In `apps/web/app/runtime-provider.tsx`
the Bearer token was held only in React state (`useState("")`) and never
persisted. On refresh it reset to blank, so `refreshConversations()` fired with
no `authorization` header. The gateway's `AllowAnonymous` path then resolved the
request to the built-in `dev-user` identity, while the stored conversation
belonged to the minted-token identity (`emp-42`). `listConversations` is scoped
to the verified `user_id`, so it returned an empty list — the history
"disappeared" because the browser had silently become a different user.

## Fix
1. **Persist the token** — lazy-init `token` from `localStorage["cocola_token"]`
   and mirror every change back (removing the key when blank). Identity now
   survives a refresh, so the real user's conversations reappear in the sidebar.
2. **Hide the session-id input** — removed the editable session-id `<input>`
   from the top-bar debug panel in `apps/web/app/page.tsx`. The session id is
   auto-generated by the runtime (`useState(genId)`) and driven by sidebar
   clicks / "new conversation"; surfacing it was dev noise. The Bearer-token
   input remains for developer use.

## Notes
- The `user_id` is derived from the token, not a fixed constant: a minted token
  yields its subject (e.g. `emp-42`); a blank token falls back to `dev-user`
  via `COCOLA_AUTH_ALLOW_ANON=1`. Persisting the token keeps the identity stable
  across refreshes.
- `sessionId` intentionally still regenerates per mount; the user lands on a
  fresh thread and picks prior conversations from the now-visible sidebar.
